# 任务路由：L0 / L1 / L2

任务等级由变更影响决定，而不是由代码行数或模型能力决定。拿不准时上调一级；不得为了省步骤而静默降级。

| 等级 | 适用情况 | 开始前最低产物 | 实施与完成条件 |
| --- | --- | --- | --- |
| L0：直接处理 | 局部、低风险、目标明确的修复或小调整；不改变业务规则、对外接口、数据结构或权限边界。 | 在对话中复述目标、影响文件和验证方式。 | 修改后执行相关验证，说明结果与残余风险。 |
| L1：短计划后处理 | 多文件或有一定实现选择，但范围明确；不涉及重要业务行为、公共契约、架构或不可逆数据变化。 | 一份简短计划：目标、范围、关键步骤、验证方式。可留在对话或变更说明中。 | 按计划实施；如果发现范围扩大，立即升级为 L2。 |
| L2：Spec 后处理 | 业务规则、权限/安全、公共 API/协议、数据模型或迁移、跨模块架构、不可逆操作，或需求与验收标准存在实质不确定性。 | 在 `openspec/changes/<change-id>/` 创建提案与任务；按需要补设计和规格。 | 指定负责人审批后才能改业务实现；完成验证、记录结论，再按项目规则归档。 |

## L2 的最小流程

1. **澄清与提案**：说明问题、目标、非目标、影响范围、风险、验收条件与回滚/降级考虑。
2. **拆解任务**：把实施与验证拆成可核查的事项，标注依赖和未决问题。
3. **必要时补设计或规格**：涉及契约、架构、数据或复杂业务决策时，写清楚选择、替代方案和行为要求。
4. **等待审批**：审批责任应先在 `openspec/project.md` 中配置；未配置时 L2 停留在 `draft`。在指定负责人确认前，不修改 L2 涉及的业务实现；可以继续调研、补文档或准备验证方案。
5. **实施与验证**：逐项完成，记录实际验证命令/方式、结果和未覆盖项。
6. **归档**：将已接受的变更沉淀为规范或历史档案，使后续任务能理解“为什么现在是这样”。

这里的“创建”“审批”“归档”都是普通 Markdown 与正常 AI 对话行为，不要求调用 OpenSpec 官方 CLI。

## 必须升级到 L2 的信号

出现下列任一项即按 L2 处理：

- 改变用户可感知的业务规则、计费、权限、合规或安全语义；
- 改变外部 API、事件、协议、数据契约或兼容性承诺；
- 新增/修改数据库结构、数据迁移、批处理或不可逆写入；
- 跨多个边界模块调整职责、依赖方向或技术方案；
- 需求目标、影响范围、验收标准或风险仍存在关键分歧；
- 负责人明确要求留下 Spec 或进行审批。

## 共同完成标准

无论等级，交付都应能回答四个问题：改了什么、为什么这样改、如何验证、还有什么风险或未验证项。L2 额外回答：谁审批、规格在哪里、最终沉淀在哪里。
