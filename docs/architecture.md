# 架构与边界

## 目标

基座为团队提供一致的工作默认值，而不侵入业务代码、不替换现有研发流程。高级模型可以自行完成大量编码工作，但仍需要稳定的项目上下文、清晰的任务边界和可复查的变更说明；这些内容应随业务仓库版本化，而不是依赖某位成员的个人习惯。

## 四层组成

| 层 | 放置位置 | 职责 |
| --- | --- | --- |
| 项目规则 | 业务仓库根目录 | 规定阅读顺序、L0/L1/L2 路由、审批点和完成标准。它是三端共同的事实来源。 |
| 项目上下文 | `openspec/project.md`、`openspec/roadmap.md` | 保存长期事实、架构约束、当前优先级与已知风险；只在确有事实变化时更新。 |
| L2 变更档案 | `openspec/changes/` 与 `openspec/specs/` | 保存高影响变更的提案、设计、任务、验收与归档结果。 |
| App 适配层 | `CLAUDE.md`、`.cursor/rules/`、Codex 插件/Skill | 将同一套项目规则以各 App 能识别的形式暴露出来；不重新定义业务规则。 |

业务团队可以额外维护一个扩展清单，列出推荐的领域 Skill、MCP 或工具说明。它只提供发现入口，不自动安装、启用、调用或授权任何外部能力。

## 运行方式

1. 成员在已初始化的业务仓库中正常发起编码对话。
2. AI 先读取项目规则和与任务相关的上下文，再按风险选择 L0、L1 或 L2。
3. L0/L1 在对话和代码变更中完成；L2 先形成可审阅的文档，再等待指定负责人确认。
4. 完成时交代影响范围、验证结果与尚未验证的风险；L2 同时留下可追溯档案。

因此，`init` 是项目接入动作，不是成员每次任务的前置命令。

## OpenSpec 的采用方式

L2 借鉴 OpenSpec 的“先说明变更，再实施，再归档”结构，但由 AI 在普通对话中直接维护 Markdown 工件：`proposal`、`tasks`、必要时的 `design` 与 `spec`。这些工件与 OpenSpec 的思路兼容，**不依赖官方 OpenSpec CLI**，也不要求每个任务都进入完整 Spec 流程。

没有 CLI 时，模型负责语义判断和文档编写；项目模板负责固定目录、最小字段与审批点。格式不一致时，以模板和人工审阅为准。

## 明确边界

- 不生成、替换或侵入业务实现；基座只提供规则、模板与适配器。
- 不限制成员已经安装的 Skill、MCP、插件或本地工具，也不为其背书。
- 不自动读取凭据、连接外部系统或执行有副作用的工具。
- 不把任何 App 插件视为强制执行器；规则能够被成员和评审看见、理解和复查才是第一优先级。
- 不包含运行时、全局命令、CI、GitHub 门禁或其他额外基础设施。
