# 发布与更新策略

## 发布对象

一个基座仓库维护三类可版本化内容：

| 内容 | 更新影响 | 原则 |
| --- | --- | --- |
| 核心模板 | 新项目的规则、L2 目录与上下文骨架 | 以可读、稳定和最小化为先。 |
| App 适配器 | Codex、Claude Code、Cursor 的工作流提示与入口 | 只改变对应 App 的辅助体验，不改变业务仓库事实。 |
| 文档与示例 | 接入、路由和升级说明 | 与模板版本同步说明。 |

每次发布应有明确版本号、变更摘要、兼容性说明与升级影响。发布仓库可以托管在团队 GitHub 组织中，作为模板和各 App 插件的共同来源。

本仓库的 `VERSION`、`release-manifest.json` 和 `scripts/build-release-assets.mjs` 只服务基座维护者：后者把 Canonical 模板打包到 Codex/Claude 与 Cursor 的发布目录，并检查三端镜像、版本和哈希是否一致。它不是成员项目的 Runtime、CLI 或日常工作流。

## 更新原则

1. **新项目取最新稳定版**：初始化时复制当期模板，之后项目规则随业务仓库一起版本化。
2. **既有项目显式升级**：先比较模板差异，再由项目负责人选择接受哪些规则、模板或适配器变化。
3. **插件更新不改业务仓库**：插件可以按各 App 的原生机制更新自身工作流，但不得静默覆盖 `AGENTS.md`、项目上下文、L2 档案或业务代码。
4. **项目事实优先**：若通用基座与既有项目事实冲突，先修订升级方案，不直接覆盖项目规则。
5. **可回退**：保留上一个稳定版本和变更说明；出现兼容性问题时回到已验证版本，再决定修复方案。

“无感”指成员日常编码不需要重复配置；不指对项目规则进行无提示、不可审阅的覆盖。

## 推荐发布节奏

1. 在试点项目验证模板和三端适配。
2. 汇总真实的误用、文档漂移和接入障碍，只做必要改动。
3. 发布带摘要的稳定版本，并让新项目使用该版本。
4. 对既有项目提供差异说明和升级建议，由项目负责人确认后合入。

高频业务知识不应被塞进通用基座；它应由各业务团队在自己的项目上下文或可选扩展中维护。这样，基座升级不会意外改变业务决策。

## 维护者发布前检查

```bash
node scripts/build-release-assets.mjs --write
node scripts/build-release-assets.mjs --check
```

`--write` 只同步本仓库受管的发布镜像；发布前仍应人工审阅 diff、更新 `VERSION` 与 Changelog。V1 不配置 GitHub Actions、CI Gate 或业务仓库门禁。

## 版本变更的判断

- 只改措辞、示例或文档：补丁版本。
- 新增兼容的模板字段、工作流入口或 App 适配：小版本。
- 改变 L2 文件结构、审批语义或已有规则含义：大版本，并提供迁移说明与人工确认点。
