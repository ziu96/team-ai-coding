# 升级团队 AI Coding 基座

插件更新只改变辅助工作流；项目内规则升级必须是一次可审阅、可确认的变更。

## 安全边界

- 当前包内快照位于 `../../templates/project`。升级到新版本时，用户必须提供明确的新版 tag、release 或 checkout；不要猜测“最新版本”。
- 不 clone、下载或推断来源路径。
- 未对精确路径获得确认前，不覆盖、删除、重命名、暂存、提交、推送、安装、启用或禁用任何内容。
- 项目已有规则、上下文与业务专有补充属于项目所有者；新版基座没有权力抹除它们。

## 流程

1. 确认目标仓库与新版模板来源，验证 `<new-template-source>/templates/project` 可读。
2. 读取当前项目规则与 Spec 文件，比较其记录的版本和新版模板；不要仅凭时间戳判断。
3. 按文件列出：安全新增、需人工合并、应保留的本地内容、可能废弃但绝不自动删除的内容。
4. 解释每项对 L0/L1/L2 路由、L2 批准边界和三端适配的影响；任何歧义都作为阻塞项。
5. 展示最小、排序后的升级计划，等待用户对路径与冲突处理的明确确认。
6. 确认后只实施已批准的新增或编辑，回读结果并报告保留、跳过和后续人工事项。

## 输出

优先给出差异摘要和未决决定，而不是笼统宣称“已升级”。比较无法安全进行时，说明缺失的来源或目标信息并停止。
