# 团队 AI Coding 基座

本文件是 Codex、Claude Code、Cursor 共同遵循的项目级入口。它定义高质量默认路径，不替代团队成员已有的 Skills、MCP 或工具设置。

## 每个任务先分级

先阅读与当前任务有关的 `openspec/project.md`；涉及现阶段优先级、范围取舍或已知风险时，再阅读 `openspec/roadmap.md`。

| 级别 | 适用情形 | 必须动作 |
| --- | --- | --- |
| L0 | 范围明确、低风险、无接口/数据/架构变化的微小修复、文档或局部重构 | 直接实施并执行针对性验证。 |
| L1 | 已知实现路径的常规功能或多文件改动 | 先给出简短计划、影响面和验证方式；再实施与验证。 |
| L2 | 新能力、公共接口、权限/安全/隐私、数据迁移、架构或依赖变化、破坏性变更，或需求/风险不清楚 | 必须进入 `openspec/AGENTS.md` 的 Spec 流程。获得明确人工批准前，不得实施业务代码或配置变更。 |

无法可靠判断时按 L2 处理。用户的显式风险判断可以补充任务背景，但不能把明显的 L2 风险降为 L0/L1。

## 共同工作约束

1. 先说明本次任务的级别和成功标准，再开始非琐碎改动。
2. 只修改完成任务所需的内容；不要顺手重构、升级依赖或改变无关行为。
3. L1 的计划应简短：目标、主要改动、验证方式即可；不要把普通任务仪式化为长方案。
4. L2 的“批准”只能来自明确的人类确认，不能由 AI 推断、补写或伪造。批准记录保留在该 change 的 proposal 中。
5. 完成前执行与风险相称的验证。无法执行时，明确说明未验证项、原因和潜在影响。
6. 不自动更新 `project.md` 或 `roadmap.md`。只有项目事实或优先级确实变化、且任务明确包含维护上下文时才更新它们。
7. `.ai-team/extensions.md` 只记录团队推荐的可选能力；不自动安装、启用、调用任何 Skill、MCP、插件或外部服务，也不限制成员已拥有的工具。

## 交付要求

交付时说明：完成了什么、验证结果、是否产生/更新了 Spec、以及仍需人工决定的事项。不要把“已阅读规则”当作交付内容。
