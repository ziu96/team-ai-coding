# L2 Spec 工作流

本目录只服务 L2 任务。它使用与 OpenSpec 一致的“提案—批准—实施—验证—归档”产物结构，但 V1 不依赖官方 OpenSpec CLI。

## 何时进入这里

根 `AGENTS.md` 判定为 L2 时，开始实施前必须阅读本文件。不要因为已有聊天计划就跳过 Spec。

## 创建 change

1. 为变更创建清晰的 kebab-case 标识，例如 `add-member-invite`。
2. 在 `openspec/changes/<change-id>/` 基于 `_template/` 创建：
   - `proposal.md`：问题、目标、范围、非目标、影响、风险、验收标准和回滚方案；
   - `tasks.md`：可验证的实施任务；
   - `design.md`：仅在方案、数据、接口或架构需要说明时创建；
   - `spec.md`：要新增或修改的、可验证的需求。
3. 先将 proposal 状态写为 `draft`，向仓库 Owner、受影响模块维护者或用户指定的批准主体请求审批。

## 批准门槛

- 先读取 `project.md` 的“L2 审批责任”。该段未配置时，默认批准主体为仓库 Owner 或受影响模块维护者；额外审批人与审批渠道仅在项目明确配置时适用，不能因此阻塞初始化、draft 或后续审批。
- 拥有相应项目或模块决策权的用户明确确认，或可追溯的 PR 审核，均可作为人工批准。若当前用户是否有此权限不清楚，只在请求 L2 实施批准时询问，不要在初始化阶段阻塞。
- 未获得明确人工批准前，不得实施业务代码、数据、配置或对外接口变更。
- proposal 顶部的“状态”是唯一状态字段。批准后把它改为 `approved`，同时在批准记录中填写批准人、批准时间和批准依据。没有这些事实时保持 `draft`。
- 审批意见要求修改时，先修订 Spec，再次请求批准；不要把“正在讨论”误写为“已批准”。

## 实施、验证和归档

1. 仅实施批准范围内的 tasks；范围变化时回到 proposal 更新并重新确认。
2. 完成每项 task 后记录对应验证证据。
3. 在 `validation.md` 汇总测试、人工检查、未验证项及结果。
4. 验收标准满足后，把 change 移至 `openspec/changes/archive/YYYY-MM-DD-<change-id>/`。
5. 若形成长期有效的需求结论，同步维护 `openspec/specs/`；不要把临时讨论当作长期规范。

## 产物质量

- 需求写成可观察、可验证的结果，避免“优化”“更好”“尽量”等不可检验表述。
- 方案应说明关键取舍和失败/回滚路径，不要求为简单改动制造设计文档。
- 任务必须可完成、可复核；不要把“让 AI 实现”当作 task。
